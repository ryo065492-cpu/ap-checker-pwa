# 出面・作業日報・作業員名簿アプリ 引継ぎ資料

更新時点: 2026-07-14

## 1. プロジェクト概要

建設・工事現場向けに、以下の業務をスマートフォン中心で行えるアプリを検討している。

- 作業員情報の登録・更新
- 現場情報の登録・更新
- 作業員名簿の作成とPDF出力
- 紙の作業日報を元にした日報入力
- 日報データを元にした月次作業日報の作成
- 日報データを元にした現場別・月別の出面表作成
- 将来的な請求・清算資料への拡張

背景として、現在は自宅にあるPCでの作業が中心で、出先から名簿送付・修正・集計ができずレスポンスが悪い。スマートフォンで現場対応できることが必須条件。

## 2. 確定している業務要件

### 2.1 作業日報

- 紙の日報は当日に提出し、原本を自社へ持ち帰る運用を当面残す。
- 紙の日報を見ながら、後からアプリに入力する。
- 日報データを月次作業日報と出面表のソースデータにする。
- 日報は現場単位で管理する。
- 初期版の入力項目は以下。
  - 日付
  - 現場
  - 作業内容
  - 人工
  - 残業
  - 備考
- 人工・残業は0.5単位を基本とし、1.0単位も選びやすくする。
- 勤務時間は `人工 × 8時間` で計算する。
- 日常画面では当日の入力を優先し、月内一覧は別画面で確認する。

### 2.2 出面表

- 現場別・月別で作成する。
- 1行は現場全体の日別合計。
- 日報データから人工・勤務時間・残業を集計する。
- 月末など決まったタイミングで清算・提出用に確認、出力する。
- 日常的に出面表へ直接入力する運用にはしない。

### 2.3 作業員名簿

- 元請へ提出する。
- 作業員情報は作業員マスターとして管理する。
- 現場情報は現場マスターとして管理する。
- PDF作成時に「現場」と「提出する作業員」を選択する。
- 入場日、役割、送り出し教育日、受入教育日などは、作業員マスターではなく「現場別作業員情報」として管理する。
- 元請指定様式に合わせてPDF化する。
- マスター管理画面から通常の登録・修正ができる。
- PDF作成中の不足チェックからも、該当するマスターまたは現場別情報を修正できる。

## 3. 初期版で扱わない項目

以下は将来追加を想定し、初期の日報入力から除外する。

- 車両
- 距離数
- 深夜残業
- 高速代
- 材料費
- 請求金額
- 請負残額
- 消費税
- 清算金額
- 請求書作成

入力項目として追加するだけなら比較的容易。ただし、集計・PDF・請求計算へ反映する場合は別途業務ルールが必要。

## 4. 情報設計

### 4.1 作業員マスター

現場が変わっても使い回す、人に紐づく情報。

候補項目:

- 氏名
- フリガナ
- 所属会社
- 職種
- 電話番号
- 緊急連絡先
- 生年月日
- 住所
- 健康診断日
- 健康保険
- 年金
- 雇用保険
- 資格・教育

正式項目は、実際の作業員名簿様式を元に確定する。

### 4.2 現場マスター

現場に紐づく情報。

候補項目:

- 現場名
- 元請
- 提出様式
- 現場住所
- 工期
- 現場担当者
- 状態

### 4.3 現場別作業員情報

「この作業員が、この現場に入る時」にだけ必要な情報。

- 現場ID
- 作業員ID
- 入場日
- 役割
- 送り出し教育日
- 受入教育日
- 現場用備考

### 4.4 日報データ

- 日付
- 現場ID
- 作業内容
- 人工
- 残業時間
- 備考
- 勤務時間は保存値ではなく、原則として人工から計算

### 4.5 出力物

- 月次作業日報
- 現場別・月別の出面表
- 元請指定様式の作業員名簿PDF

## 5. UI・UXの確定方針

- スマートフォン運用が必須。PCは補助。
- ホーム画面から各業務へ移動する。
- 「マスター管理」と「書類作成」を明確に分ける。
- 日報入力画面では、その日の情報だけを表示する。
- 月累計は日報一覧・出面表で表示する。
- 画面上の数値は、`人工合計`、`勤務時間合計`、`入力済み日数`のように意味が明確なラベルにする。
- 1〜31日を入力画面へ大量表示しない。
- 一覧は別画面にまとめる。
- 長いフォームを常時表示せず、項目カードから編集画面を開く。
- 人工・残業はプリセット、±操作、直接入力を用意する。
- タップ領域、余白、コントラストを現場利用に耐えるサイズにする。
- AI的な説明文、過剰なカード、曖昧な指標、仮想モニター件数の表示は入れない。
- 「説明を読まないと分からないUI」は不合格。
- 見た目だけでなく、何を登録し、何を組み合わせ、どこで出力するかが明確であること。

## 6. 却下・修正されたUIパターン

Codexや後続担当が同じ方向へ戻さないための注意事項。

- 「かんたん／詳細」モード: 意味が分かりにくいため却下。
- 日報入力画面に1〜31日のチップを並べる: 30日分になると見づらいため却下。
- 日報入力画面に月累計を表示する: 入力文脈と合わないため却下。
- `月人工`、`勤務H`、`入力日`など曖昧な略称: 却下。完全な日本語ラベルを使う。
- 初期版から車両・距離・深夜残業を入れる: 項目過多のため除外。
- 名簿作成を単なる4ステップ提出フローとして見せる: マスター構造が不明瞭になるため修正。
- 「仮想モニター500件／1000件」などの文言をUIに表示する: 不要。
