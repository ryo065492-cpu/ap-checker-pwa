# 本番セキュリティ・個人情報取扱要件

- 更新日: 2026-07-16
- 対象: 依頼者が利用する新規本番アプリ
- 対象外: 既存HTML、`poc/roster-ui/`、`poc/roster-pdf/`
- データ管理主体: 依頼者企業
- 受託開発者: 諒さん
- 状態: 基本方針は確定。保持期間など依頼者判断が必要な項目はP0ゲートで承認する

この文書は、本番セキュリティに関して `docs/PROJECT_HANDOFF_JA.md`、`docs/CODEX_HANDOFF_JA.md`、`docs/ARCHITECTURE_AUDIT_JA.md` の古い候補記述より優先する。

## 1. セキュリティ目標

1. 本番データを閲覧・管理できるのは、依頼者が許可した利用者だけとする。
2. 諒さんは、納品後の本番DB、認証管理、秘密鍵、ログ、監視、バックアップ、生成PDF、本番デプロイへアクセスできない状態にする。
3. 開発、テスト、レビュー、サポートでは匿名の合成データだけを使用し、本番データを複製しない。
4. 初期本番は依頼者1社専用とし、自社情報1件と自社作業員だけを扱う。協力会社作業員、所属会社選択、複数会社管理、組織切替は実装しない。
5. セキュリティは画面の非表示だけに依存せず、認証、DB認可、最小権限、暗号化、監査、運用手順を重ねて実現する。

## 2. P0: 実データ投入前の必須ゲート

次のP0が1件でも未完了なら、実データを登録せず、本番公開もしない。

| Gate | 合格条件 | 主な証跡 |
| --- | --- | --- |
| P0-1 所有権 | GitHub、ホスティング、Supabase、ドメイン、請求、監視、バックアップを依頼者が所有する | 所有者・メンバー一覧 |
| P0-2 データ最小化 | 各項目の利用目的、帳票上の必要性、閲覧者、保持期間、削除方法を依頼者が承認する | データ項目台帳 |
| P0-3 認証・権限 | 招待制、個人別ID、MFA、役割表、退職・異動時の失効手順が動作する | 権限表、認可テスト |
| P0-4 委託先 | クラウドの契約、DPA、データ処理地域、再委託先、事故通知、終了時削除を確認する | 委託先台帳、契約記録 |
| P0-5 保持・削除 | 作業員、現場、ドラフト、監査ログ、バックアップごとの保持期間を決定する | 保持・削除表 |
| P0-6 事故対応 | 依頼者側の責任者、連絡先、封じ込め、調査、報告判断、復旧手順を定める | インシデント手順書 |
| P0-7 技術検証 | 未認証・権限外アクセス拒否、秘密・PII混入防止、脆弱性検査、復元試験に合格し、重大・高リスクが0件である | CI、試験記録、復元記録 |
| P0-8 引渡し | 諒さんの本番権限を削除後、依頼者が秘密情報を再発行し、依頼者だけで停止・復旧できる | 引渡しチェックリスト |

## 3. 収集する情報を最小化する

初期版で収集できるのは、承認済みの作業員名簿に必要な次の情報だけとする。

- 自社情報1件
- 自社作業員の帳票必須情報
- 現場情報
- 現場別作業員情報
- 資格・教育の帳票必要情報
- 認証ユーザーと最小限の監査メタデータ

次の情報は初期版では収集しない。

- マイナンバー
- 健康診断日、健康診断結果、病歴
- 保険者番号、被保険者の記号・番号の全文
- 資格証明書や本人確認書類の画像
- GPS、顔写真、生体情報
- 協力会社作業員の情報
- 利用目的が決まっていない自由記述
- 生成PDF、PDFスナップショット、PDF履歴

正式帳票上の必要性、取得根拠、閲覧権限、保持期間が改めて承認されない限り、禁止・保留項目をschema、フォーム、APIへ追加しない。年齢など再計算できる値は原則として重複保存しない。

## 4. 所有権、開発、引渡し

- 本番アカウントは依頼者が作成・所有し、諒さん名義の本番契約を使用しない。
- 開発環境と本番環境を分離する。開発、プレビュー、CIは匿名データ専用とし、本番DBの複製を持ち込まない。
- 空の本番環境を構築するために一時アクセスが必要な場合でも、実データ投入前に諒さんを全サービスから削除する。
- 権限削除後、依頼者がDBパスワード、API secret、デプロイトークン、SMTP等を再発行する。再発行した値を諒さん、Codex、チャットへ渡さない。
- 継続保守では、匿名環境へのPRまたは署名済みreleaseを提出する。依頼者の承認後、依頼者管理のCIが本番へ反映する。
- 諒さんに本番への直接merge、deploy、deploy hook、DNS、監視、バックアップ権限を残さない。

## 5. 認証と認可

- 一般公開の利用登録を無効にし、依頼者管理者による招待制にする。
- 共有アカウントを禁止し、利用者ごとに個別アカウントを発行する。
- 全利用者にMFAを必須とし、アプリとDBの双方でMFA済みセッションを要求する。
- 初期役割は `customer_admin`、`editor`、`viewer` を候補とし、最終権限表をP0で承認する。
- アカウント無効化は直ちに全セッションへ反映する。
- 無操作30分、最大12時間を初期セッション案とし、依頼者の運用確認後に確定する。
- 全テーブル、View、Function、Storageを既定拒否にし、RLSまたは同等のサーバー認可を適用する。
- 単一顧客専用でも全業務行に固定の `organization_id` を持たせ、認証ユーザーのmembershipと役割で照合する。複数組織を切り替えるUIは作らない。
- `SELECT`、`INSERT`、`UPDATE`、`DELETE`を別々に許可し、書込みは新しい値にも所有境界を適用する。
- secret keyや`service_role`をブラウザ、source map、Git、ログ、チャットへ含めない。不要なら本番アプリで使用しない。
- Supabase Authは初期版で`persistSession: false`とし、認証情報をメモリだけに保持する。再読込時は再ログインさせ、利便性のため永続化する変更は別途リスク承認する。

## 6. Web、通信、端末

- 本番はHTTPSだけを許可し、HSTS、厳格なCSP、`frame-ancestors`、`nosniff`、Referrer-Policyを設定する。
- DBのSSLを強制し、直接DB接続を必要最小限の接続元に制限する。
- 個人情報、認証トークン、氏名、現場名をURL、クエリ文字列、ファイル名へ含めない。
- 入力値を信頼せず、XSS、SQL Injection、CSRF、IDOR、過大入力、総当たりに対する検証とレート制限を行う。
- 初期版はオンライン専用とし、個人情報を`localStorage`、`sessionStorage`、IndexedDB、Service Worker Cacheへ永続保存しない。
- 解析SDK、広告SDK、session replayを初期版へ導入しない。
- 利用端末には画面ロック、OS更新、端末暗号化を求める。PDFダウンロード後の管理は依頼者の端末・文書管理規程の対象とする。

## 7. PDFとファイル

- 名簿PDFをDBと同等の個人データとして扱う。
- MVPはブラウザのメモリ上でPDFを生成して利用者へ直接ダウンロードし、サーバー、Storage、履歴へ保存しない方式を第一候補とする。
- クライアント生成で正式様式の品質を満たせない場合だけ、依頼者所有の処理環境で認証後に一時生成し、レスポンス後に削除する方式を再審査する。
- 公開bucket、恒久URL、メール自動添付、Git、レビュー画像、サポートチケットへの実PDF混入を禁止する。
- PDF生成・ダウンロード操作は、個人情報の値を含めずに監査記録する。
- PDF応答をサーバー経由にする場合は `Cache-Control: private, no-store` とし、短時間URL以外を発行しない。

## 8. ログと監査

- ログイン、ログイン失敗、アカウント無効化、権限変更、作成・変更・削除、PDF生成を監査する。
- 監査ログは「誰が、いつ、どの種類の対象へ、何を行い、成功したか」を記録する。
- 氏名、住所、電話番号、保険情報、入力本文、PDF内容、秘密情報をログへ記録しない。
- アプリ監査ログは通常利用者から更新・削除できない構造にする。
- 初期保持期間は1年を提案し、依頼者の運用・法務確認後にP0で確定する。
- エラー監視はrequest ID、エラー分類、匿名化された技術情報だけを扱う。

## 9. バックアップ、保持、削除

- 本番は自動日次バックアップを最低基準とし、初期目標をRPO 24時間以内、RTO 8時間以内とする。
- 四半期ごとに、依頼者だけがアクセスできる隔離された一時リカバリ環境へ本番バックアップを復元し、復元手順を確認する。諒さんは参加・アクセスせず、試験後は環境と復元データを確実に破棄する。
- 作業員、退職者、現場、現場別情報、ドラフト、認証アカウント、監査ログ、バックアップごとに保持期間と起算点を決める。
- 利用目的がなくなった情報は削除または匿名化し、削除実行の記録を残す。
- バックアップについても保持期限を定め、期限経過後に自然消去される時点を記録する。
- PDF、実データ入りサポート資料、開発環境への本番複製は保存しない。

具体的な保持年数は、帳票保存義務、元請契約、依頼者の労務運用を確認せずに実装者が推測しない。

## 10. 開発とCI/CD

- 本番コードは専用の非公開リポジトリで管理する。
- `main`への直接pushと、開発者単独の本番デプロイを禁止する。
- PRレビュー、型検査、lint、unit/integration/E2E、RLS拒否テスト、依存関係検査、secret scanningを必須化する。
- Preview環境へ本番秘密や本番データを渡さない。
- 依存バージョンとlock fileを固定し、重大・高リスクの脆弱性を残したまま公開しない。
- デプロイ対象のcommit SHAと承認者を記録する。

## 11. 漏えい・事故対応

依頼者は本番前に、次の流れを含む手順と連絡網を用意する。

1. 発見者から責任者へ直ちに連絡する。
2. セッション失効、鍵交換、アクセス遮断で拡大を防ぐ。
3. 証拠を保全し、対象項目、人数、期間、原因を調査する。
4. 個人情報保護委員会への報告と本人通知の要否を判断する。
5. 復旧、再発防止、判断記録、事後レビューを行う。

報告対象となる場合の期限を満たせるよう、クラウド・委託先の事故通知は契約上「直ちに」とし、公開前に机上訓練を行う。

## 12. セキュリティ受入試験

- 未認証利用者の全CRUDが拒否される。
- viewerの書込み、editorの権限変更、無効化済み利用者の全操作が拒否される。
- 合成した別organization IDを指定しても、読取り・更新・PDF生成が拒否される。
- ブラウザbundle、source map、Git履歴、ログにsecretが0件である。
- `localStorage`、`sessionStorage`、IndexedDB、Service Worker CacheにPIIが0件である。
- XSS、CSRF、IDOR、入力長、レート制限、認証回避の検査に合格する。
- PDF、URL、HTTP cache、ログ、分析へPIIが意図せず残らない。
- バックアップ復元、アカウント即時失効、秘密情報ローテーションに成功する。
- 重大・高リスクの脆弱性が0件である。

テスト証跡には匿名データだけを使用する。

## 13. P0で依頼者の決定が必要な項目

- 初期利用者数と各利用者の役割
- 現場ごとの閲覧範囲が必要か
- 利用端末が会社管理端末かBYODか
- 各データの保持期間
- 退職者の削除・匿名化時期
- ドラフトの保存期間
- PDFダウンロード後の保管・送付方法
- 事故時の責任者と連絡先
- 脆弱性対応、権限棚卸し、復元試験の担当者
